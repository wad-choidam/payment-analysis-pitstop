# 결제 불일치 분석 가이드

## 개요

BPOS/CPOS(iOS), APOS(Android) ↔ Eximbay 단말기 간 결제 불일치 발생 시 원인을 파악하기 위한 가이드입니다.

> 이 문서의 내용은 분석 도구의 `/guide` 페이지에서도 시각적으로 확인할 수 있습니다. 도구 실행 후 헤더의 "가이드" 버튼을 클릭하세요.

## 지원 POS 종류

| POS 종류 | OS | ptxId 접두사 | 엑셀 포맷 |
|---|---|---|---|
| BPOS | iOS | `BPOS-` | `raw.detailLog` 텍스트 로그 |
| CPOS | iOS | `CPOS-` | `raw.detailLog` 텍스트 로그 |
| APOS | Android | `APOS-` | 각 행이 하나의 이벤트 (구조화된 데이터) |

## 대상 단말기

| 단말기 | 직전 거래 조회 | 중복 결제 체크 | 단말기 로그 | 포스 로그 |
|--------|:---:|:---:|:---:|:---:|
| Eximbay | O | O | O | O |
| KIS | X | X | X | O |
| KOCES | X | X | X | O |

결제 불일치 분석은 **Eximbay 단말기 사용 매장**에서 주로 확인합니다. KIS/KOCES는 단말기 로그가 없어 포스 로그만으로 판단하며, 정확한 원인 확인이 어렵습니다.

## 분석 흐름

### 1. 알림 확인

Slack `#결제데이터-불일치-확인` 채널에서 `[결제데이터 불일치 확인 필요건]` 알림을 확인합니다.

- **BPOS**: 알림의 txId에 `BPOS-` 포함 (예: `txId: BPOS-b7a0b1d8-...`)
- **CPOS**: 알림에 `clientVersion` 표기 (예: `clientVersion: 1.2.2`)

### 2. 로그 수집

- **포스 로그**: 알림 스레드에 엑셀 파일(`{매장명}_{orderSeq}.xlsx`)로 자동 첨부됨
- **단말기 로그**: 페이서비스 팀이 시리얼번호를 메모로 남기고, vhq.vficonnect.com에서 다운로드 후 공유

### 3. 로그 입력

이 도구(`pnpm dev`)를 실행한 후:

1. 포스 로그를 좌측 입력창에 입력 (아래 3가지 방법 중 택 1)
   - **엑셀 파일 업로드** (권장): Slack 스레드의 `.xlsx` 파일을 드래그앤드롭하거나 "파일 선택"으로 업로드 → detailLog 자동 추출
   - **텍스트 붙여넣기**: 포스 로그의 `detailLog` 부분을 직접 붙여넣기
   - **txt 파일 업로드**: `.txt` 파일을 드래그앤드롭
2. 단말기 로그를 우측 입력창에 붙여넣기 (또는 `.txt` 파일 드래그앤드롭)
3. "분석 시작" 클릭

> **CPOS 엑셀 파일 구조**: 시트 "이벤트 로그"의 `raw` 컬럼 안 JSON의 `detailLog` 필드가 포스 로그입니다. 각 행이 하나의 결제 시도 세션에 해당합니다.

## 포스 로그 형식

포스 로그는 JSON 내 `detailLog` 필드에 담겨 있으며, `#HH:mm:ss.SSS` 형식의 타임스탬프로 구분됩니다.

```
#14:46:53.615 [전송] 승인 접속 시도
#14:46:54.059 [수신] 단말기 접속 성공
#14:46:54.061 [전송] 직전 거래 데이터 요청
#14:46:54.106 [수신] 승인 전문 수신 - CPOS-xxx, 결제 성공
#14:46:54.108 [전송] 카드 결제승인 요청 전문 전송 - CPOS-yyy
```

## 단말기 로그 형식

`YYYY-MM-DD HH:mm:ss:SSS` 형식이며, `[version]` prefix가 붙을 수 있습니다.

```
[1.0.127] 2024-11-19 14:46:53:903 AgentSocketServer: CLIENT IP/192.168.0.14
[1.0.127] 2024-11-19 14:46:53:907 <REQUEST>: [POS -> T650P] INQUIRY_LATEST_TRANSACTION(990001)
```

`AgentSocketServer: CLIENT IP`가 POS → 단말기 연결 요청의 시작점입니다.

## POS ↔ 단말기 통신 흐름

```
POS → 단말기    연결 요청
POS ← 단말기    연결 성공
POS → 단말기    직전 거래 데이터 요청 (중복 결제 체크용)
POS ← 단말기    직전 거래 데이터 응답
POS             중복 결제 확인 (ptxId 비교)
POS → 단말기    카드 결제 승인 요청
                    단말기 → VAN   승인 요청
                    단말기 ← VAN   승인 응답
POS ← 단말기    승인 결과 응답
```

## 주요 불일치 유형

### 1. 타이머 만료

포스에서 승인 요청 후 일정 시간 내 응답이 없으면 타이머가 만료되어 접속을 종료합니다.
이 사이에 단말기가 VAN과 단독으로 승인을 완료하면 결제 불일치가 발생합니다.

**포스 로그 특징**: `타이머 3초 미응답으로 접속 종료`, `마지막 승인데이터 요청에 대한 응답실패`

### 2. 단말기 수신 불가 (9999)

단말기가 다른 작업 중이거나 결제 진입 불가 화면에 있을 때 POS 요청을 받지 못합니다.
POS는 실패로 처리하지만, 단말기에서 이후 단독으로 결제를 진행할 수 있습니다.

**포스 로그 특징**: `[수신] 승인 전문 수신 - , 단말기가 수신 가능한 상태가 아님`
**단말기 로그 특징**: `res: 9999`, `Transaction Skipped! NOT AVAILABLE`

### 3. 충전독 접촉/분리

영업 중 충전독에 접촉/분리 시 통신 불량이 발생하여 결제 불일치가 발생할 수 있습니다.

**단말기 로그 특징**: `ChargingStatusReceiver: ACTION_POWER_DISCONNECTED`

### 4. PtxID 불일치 (동기화 실패)

승인 과정 중단 후 직전 거래 조회 시, 요청한 ptxId와 단말기의 직전 거래 ptxId가 다른 경우입니다.

**포스 로그 특징**: `승인정보 동기화 실패 (PtxID 불일치)`

### 5. EOT 비정상

단말기에서 결제는 성공했으나 POS와의 정상 종료 절차(EOT)가 완료되지 않은 경우입니다.

**단말기 로그 특징**: `ResultActivity: [!] EOT ABNORMAL | Print itself`

### 6. IC카드 리딩 실패 (CX01/CX06)

단말기에서 IC카드 읽기에 실패하여 MSR 모드로 전환하거나 재시도하는 경우입니다. 보통 여러 차례 실패 후 성공하며, 중복 결제가 아닌 경우가 대부분입니다.

**단말기 로그 특징**: `CardReaderPresenter: [!] FAIL | ICC CARD READ | code: CX01` 또는 `CX06`

### 7. 프린터 오류 (9959)

단말기 프린터 오류로 인해 매장에서 결제가 안 된 것으로 오인하여 재시도하는 경우입니다.

**단말기 로그 특징**: `res: 9959`

### 8. VAN 통신 타임아웃

단말기에서 VAN사로 승인 요청을 보냈으나 응답을 받지 못한 경우입니다. 네트워크 장애가 원인이며, 결제 자체가 실패합니다.

**단말기 로그 특징**: `SocketTimeoutException`

### 9. 강제 취소 타임아웃 (9989)

VAN 통신 불가 상태에서 강제 취소 요청도 타임아웃된 경우입니다.

**단말기 로그 특징**: `res: 9989`, `CARD_FORCE_CANCEL`

## 결제 시도 그룹화

분석 도구는 로그를 **N차 결제 시도**로 자동 그룹화합니다.

새로운 결제 시도의 시작점:
- POS 로그: `[전송] 승인 접속 시도` (단말기 연결 요청)
- 단말기 로그: `AgentSocketServer: CLIENT IP` (POS 연결 수신)

각 시도별로 VAN 승인 전송 여부, 0000 응답 여부, POS 수신 여부를 추적합니다.

### 승인 성공 판정 방식

포스 로그에서 `[수신] 승인 전문 수신 - CPOS-xxx, 결제 성공` 형식은 직전거래 조회 응답과 실제 승인 결과가 동일합니다. 이를 구분하기 위해 **문맥 기반 + ptxId 일치 판정**을 사용합니다:

- `카드 결제승인 요청 전문 전송` **이후에** 나온 성공 응답만 실제 승인 성공으로 판정
- 그 이전의 성공 응답은 직전거래 조회 응답으로 간주
- 승인 요청 이후에 나온 응답이라도 **ptxId가 현재 승인 요청의 ptxId와 일치할 때만** 실 승인으로 카운트. 과거 거래의 지연 응답이나 취소 후 재조회된 직전거래 응답이 실 승인으로 오판되는 문제를 방지합니다.

### 포스 로그만으로 분석할 때

단말기 로그가 없는 경우(만료 등), POS 측 승인 성공을 VAN 승인 성공으로 간주하여 분석합니다. 단, 결론에 "단말기 로그 없이 분석한 결과"라는 안내가 추가됩니다.

## Android (APOS) 분석

안드로이드 엑셀은 iOS와 구조가 완전히 다릅니다. 각 행이 이미 파싱된 **하나의 이벤트**이며 `raw.response`에 resultCode/resultMessage가 담겨 있습니다.

### 플랫폼 감지
- 엑셀의 `osType` 필드가 `"AOS"`이면 Android로 분기 (`src/parser/excelLogParser.ts`).
- Android면 `parseAndroidExcelRows()`로 엑셀 행 → `LogEntry[]` 직접 변환 (텍스트 파싱 불필요).

### 이벤트 매핑 (data2 → 정규화 이벤트명)

| data2 (엑셀) | 정규화 이벤트 | 판정 조건 |
|---|---|---|
| 카드 결제 승인 클릭 | 승인 요청 | 항상 |
| 카드 리딩 요청 | 카드 리딩 요청 | 항상 |
| 카드 데이터 응답 | 승인 성공 / 단말기 수신 불가 (9999) / 승인 실패 / 사용자 취소 / 타이머 만료 | `response.resultCode` 및 `resultMessage` 기반 |
| 이전 카드 데이터 클릭 | 직전거래 조회 클릭 | 항상 |
| 이전 카드 데이터 요청 | 직전거래 조회 요청 | 항상 |
| 이전 카드 데이터 응답 | 직전거래 응답 (성공/단말기 수신 불가/실패) | resultCode 기반 |

### 시도 그룹화 (Android 전용 경계)

iOS는 `단말기 연결 요청`이 새 시도의 시작점이지만, Android는 이 이벤트가 없습니다. 대신 **같은 슬라이스 안에 이미 `승인 요청`이 있는 상태에서 새로운 `승인 요청`이 나오면 새 시도로 간주**합니다. iOS는 항상 `단말기 연결 요청`이 먼저 경계를 만들기 때문에 이 조건에 걸리지 않습니다.

### 실 승인 판정

iOS와 동일한 `hasPosApprovalSuccess()` 로직을 재사용합니다:
- `승인 요청` 이후에 `승인 성공` 이벤트가 있으면 실 승인 (data2 "카드 데이터 응답" + resultCode=0000)
- `직전거래 응답 (성공)` 이벤트는 ptxId가 현재 `승인 요청`의 ptxId와 일치할 때만 실 승인으로 카운트

### 실제 사례 패턴 분류 (PLF-2575 에픽 29건 학습 기반)

모든 AOS 결제 불일치 사례는 다음 5가지 패턴 중 하나입니다. 대부분 "중복 결제 없음 / 정상 처리"로 결론지어지며, **예외는 1건(PLF-3097)뿐**.

#### 1️⃣ 표준 패턴 (약 45%)
```
카드 결제 승인 클릭 → 단말기 VAN 승인 (0000)
→ 서버 카드 데이터 응답 9999 "단말기 수신 불가"
→ 이전 카드 데이터 요청 → 응답 0000 (동일 ptxId)
→ 복원 완료
```
대표 티켓: PLF-5872(킴보), PLF-5323, PLF-4922, PLF-4768, PLF-4367, PLF-4335

#### 2️⃣ 취소 요청 개입 변형 (약 20%)
```
1차 승인 성공 → EOT ABNORMAL → 단말기 자체 영수증
→ POS가 뒤늦게 AGENT_CARD_READING_CANCEL 전송
→ 단말기는 이미 승인 완료라 9999 응답
→ Transaction Skipped! NOT AVAILABLE
→ 직전거래 요청으로 복원
```
대표 티켓: PLF-5663, PLF-4534, PLF-4530, PLF-3669, PLF-3604, PLF-3490, PLF-3395, PLF-3233

#### 3️⃣ 복수 시도 / 혼합 결제 (약 15%)
여러 번 시도 후 하나가 성공하거나, 카드와 간편결제가 섞인 케이스.
- **PLF-4727**: 4회 시도, ptxId 4개, 성공/실패 번갈아
- **PLF-4141**: 카드 + 간편결제 혼합 (간편결제 4회 실패)
- **PLF-3201**: 카드 승인 → 간편결제 4회 실패 → 직전거래 복원
- **PLF-3056**: IC Fallback 2회 → 3차 승인 → 4차 9999

#### 4️⃣ 사용자가 별도 취소 진행 (약 10%)
- **PLF-4379**: 승인 13초 뒤 사용자가 새 ptxId로 취소 완료
- **PLF-4340**: 취소 요청 4회 반복 후 성공
- **PLF-5532**: 결제 → 취소(CARD_CANCEL 010020) → 재시도 3단계

#### 5️⃣ 특이/예외 케이스
| 티켓 | 특이점 |
|---|---|
| **PLF-3097** | 2건 동시 결제 중 1건 복원 실패 → **포스 누락, UBMS 수동 보정** (완전 정상 아닌 유일 케이스) |
| **PLF-3577** | 카드 승인 후 현금 결제로 전환 |
| **PLF-5530** | 1건에 2개 독립 결제 (ptxId 2개, 별건) |
| **PLF-4534** | AGENT_PRINT 9959 (프린터 인쇄 충돌) |

### 분석자 공통 워크플로우

1. Slack `#noti-pos-결제데이터-불일치` 채널 알림 → Jira 티켓 자동 생성
2. 매장 시퀀스, ptxId, 응답코드 확인 (알림 본문)
3. `vhq.vficonnect.com/DeviceUploadedFiles/...txt` 단말기 로그 다운로드
4. 타임라인 분석 → Jira 댓글에 마크다운 표로 요약
5. 결론 명시 ("중복 결제 없음 / 정상 승인건")

> 최근 티켓(2026년 이후)은 Slack 알림 봇이 `{매장명}_{orderRootSeq}.xlsx` 파일도 자동 첨부합니다. 과거 티켓은 단말기 로그 URL만 제공됐습니다.

## 결론 판단 기준

실 승인 건수를 기반으로 판정합니다:
- 단말기 로그 있을 때: VAN 승인 응답(0000) 건수
- 단말기 로그 없을 때: POS 측 승인 성공 건수 (승인 요청 이후 성공 응답 기준)

| 실 승인 | 결론 |
|:-------:|------|
| 0건 | 중복 결제 아님 (승인 자체가 안 됨) |
| 1건 | 중복 결제 아님 (정상) |
| 2건 이상 | **중복 결제 의심 — UBMS 확인 필요** |

> 48건의 실제 분석 사례 중 90% 이상이 "중복 결제 아님"으로 결론지어졌습니다.

## 응답코드 참조

| 코드 | 의미 | 심각도 |
|------|------|--------|

## 핵심 용어

| 용어 | 설명 |
|------|------|
| ptxId (eximbayPtxId) | Eximbay 단말기 거래 고유 ID (`BPOS-UUID` 또는 `CPOS-UUID`) |
| VAN | 결제 대행사 (단말기가 VAN과 직접 통신하여 카드 승인 처리) |
| EOT | End of Transaction, 거래 종료 절차 |
| 직전 거래 조회 | 중복 결제 방지를 위해 이전 거래의 ptxId를 조회하는 과정 |
| 임의등록 | 중복 결제 의심 팝업에서 사장님이 수동으로 결제 처리하는 것 |
| UBMS | 결제내역 관리 시스템 (중복결제 의심 시 금액/카드번호/결제자 확인) |
